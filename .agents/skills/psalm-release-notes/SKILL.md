# psalm-release-notes

Draft or rewrite release notes for a psalm-plugin-laravel version from a GitHub release URL.

## Instructions

### 1. Parse the Release URL

Extract `OWNER`, `REPO`, and `TAG` from the URL argument (e.g. `https://github.com/psalm/psalm-plugin-laravel/releases/tag/v4.7.0` â†’ owner=`psalm`, repo=`psalm-plugin-laravel`, tag=`v4.7.0`).

### 2. Fetch Existing Draft and Previous Tag

Run in parallel:

```bash
# Get current draft/published release notes (may be auto-generated by release-drafter)
gh release view TAG --repo OWNER/REPO --json body,tagName,name

# Get previous release tag
gh release list --repo OWNER/REPO --limit 10 --json tagName,createdAt
```

### 3. Get Merged PRs

```bash
# Find the date of the previous release
# Then list all merged PRs between previous tag and current tag
gh pr list --repo OWNER/REPO --state merged --search "merged:>PREV_DATE" --json number,title,labels,body,mergedAt --limit 100
```

Filter out PRs with label `release:skip-changelog`.

Group by label:
- `release:feature` â†’ **Features**
- `release:fix` â†’ **Fixes**
- `release:internal`, `release:typo`, `chore` â†’ **Internal** (omit from notes unless user-relevant)

If a PR title alone is unclear, read its body briefly â€” but only go deeper into code if the type/intent is genuinely ambiguous.

### 4. Write the Release Notes

Apply these rules:

**Structure:**
```
<1-2 sentence theme intro>

### Features
- <one sentence, action verb, present tense> (#NNN)

### Fixes
- <one sentence, action verb, present tense> (#NNN)

**Full Changelog**: https://github.com/OWNER/REPO/compare/PREV_TAG...TAG
```

The **theme intro** is 1-2 sentences summarizing the release focus. Derive it from the release name (e.g. "v4.7.0 - Eloquent Builder" â†’ focus is Eloquent Builder improvements). If the release has no name/theme, infer one from the dominant area of the merged PRs. Keep it factual, no marketing language.

**Writing rules:**
- One line per change. Start with an action verb: `Add`, `Fix`, `Support`, `Remove`, `Improve`.
- Be specific: say *what* changed and *why it matters* in one phrase. Bad: "Fix stub issue." Good: "Fix `Collection::pluck()` returning `array` instead of `Collection` for chained calls."
- No contributor credits unless the user asks â€” this is auto-populated by GitHub release drafter on publish.
- No vague filler: no "various improvements", no "stability fixes", no "internal refactoring" unless user-impacting.
- Omit **Internal changes** section entirely unless it contains something users care about (e.g., dropped PHP support, added Psalm version requirement).
- Skip patch-level noise: style fixes, test additions, CI changes.
- If there are breaking changes, place them first under `### Breaking Changes` with a migration note.

### 5. Present to User

Output the final release notes as a markdown code block, ready to paste into GitHub.
Ask if the user wants any adjustments.
